클라이언트에서 멱등성 키를 UUID로 만드는 이유

정유정2026년 01월 20일2분이면 읽어요

멱등성 키를 UUID로 만드는 이유와, 클라이언트에서 UUID로 멱등성 키를 처리하는 방법을 정리한다.

개요

클라이언트는 서버에 동일한 요청을 여러 번 보낼 수 있다. 이 과정에서 중복된 요청이 서버에 전달되어 동일한 작업이 반복 수행되는 문제를 방지하려면 멱등성(idempotency)을 보장하는 방법이 필요하다.

개인적으로 멱등성에 대해서 깊이 고민하게 된 이유는 한 프로젝트에서 콘텐츠 관리 기능을 구현할 때 클라이언트와 서버, 그리고 외부 결제 연동 사이의 통신을 일관되게 유지하는 방법에 대해 고민했기 때문이다.

특히 이 글에서는 멱등성 키를 생성할 때 UUID(Universally Unique Identifier)를 사용하는 이유와 그 중요성에 대해 다루고자 한다. 또한, 클라이언트에서 UUID를 활용한 멱등성 키 처리 방법에 대해서도 구체적인 예시를 들어 설명한다.

멱등성이란 무엇인가?

멱등성은 시스템이나 서비스에서 동일한 작업을 여러 번 수행해도 결과(response)가 변화하지 않는 성질을 의미한다.

일반적으로 HTTP 메서드 중에서 GET, PUT, DELETE는 멱등성의 특성을 자연스럽게 갖추고 있으며, POST와 PATCH는 멱등성이 보장되지 않는 경우가 많다. 특히, POST 요청은 리소스를 생성하거나 상태를 변경하기 때문에 중복 요청 시 불필요한 작업이 이루어질 수 있어 필요한 경우 멱등성 키를 이용한다.

멱등성 키는 왜 필요한가?

네트워크 장애나 클라이언트의 실수로 인해 동일한 요청이 여러 번 서버에 도달할 때 이를 처리하기 위해 멱등성 키가 사용된다. 이를 통해 서버는 같은 작업이 중복되어 처리되는 것을 방지하고, 클라이언트는 요청에 대한 결과를 예상할 수 있다.

예시:

  1. 클라이언트가 주문을 생성하는 POST 요청을 서버에 보낸다.
  2. 요청 중 네트워크 장애로 클라이언트가 요청을 다시 보내지만, 서버는 이미 주문이 생성된 상태로 인식하고 중복된 주문을 처리하지 않는다.

이때 서버가 중복된 요청을 처리하지 않도록 보장하는 것이 멱등성 키다. 이 키는 요청이 중복되었는지 판단하는 기준이 되어, 동일한 요청에 대해 서버는 동일한 결과만 반환한다.

클라이언트에서 멱등성 키를 왜 UUID로 만들어야 하나?

멱등성 키는 꼭 UUID로 만들어야만 하지 않는다. 멱등성 키의 예시가 거의 UUID로만 되어 있어 UUID만 가능하다고 착각을 할 수 있지만, 앞서 말한 대로 멱등성 키의 역할은 동일한 요청에 대한 동일한 결과를 얻는다는 것이다. 그렇기에 클라이언트에서 요청에 대한 event를 연속적으로 발생시켰을 때 똑같은 멱등성 키를 header에 넣어야 클라이언트는 한 번만 클릭하는 케이스에 대한 원하는 예상 값을 받을 수 있다.

필수가 아니지만 UUID를 사용하는 주요 이유

  1. 고유성 보장: 1조 개의 UUID 중에 중복이 일어날 확률이 10억 분의 1로 매우 낮다.
  2. 예측 불가능성: UUID는 완전히 랜덤한 값으로 생성되어 보안성이 뛰어나다.
  3. 크기 효율성: UUID는 작은 크기로 정렬, 차수, 해싱 등 다양한 알고리즘에 사용하기 쉽고 데이터베이스에 보관하기도 용이하다.
  4. 간편성: UUID는 중앙 시스템에 등록하고 발급하는 과정이 없어서 더 빠르고 간단하게 만들 수 있다.

연속적인 HTTP 요청 시 UUID로 멱등성 키 처리 예시

클라이언트에서 멱등성 키를 UUID로 생성하여 연속적인 POST 요청 또는 PATCH 요청을 처리하는 방식은 다음과 같다.

1. 요청 전 UUID 생성

클라이언트는 요청을 보내기 전에 UUID를 생성한다. 이 UUID는 요청이 중복되었는지 판단하는 기준이 된다.

// UUID 생성 (예시 - JavaScript에서 uuid 패키지를 사용)
const requestId = uuidv4();

2. UUID를 요청에 포함

생성된 UUID는 요청의 헤더나 바디에 포함되어 서버로 전송된다. 예를 들어, HTTP 헤더에 Idempotency-Key라는 이름으로 UUID를 포함시킬 수 있다.

// POST 요청 시 멱등성 키 포함
fetch('https://example.com/orders', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Idempotency-Key': requestId,
  },
  body: JSON.stringify({
    itemId: 123,
    quantity: 2,
  }),
});

맺음말

"클라이언트에서 왜 멱등성 키는 UUID로 만드는가?"라는 포스트를 쓰게 된 결정적 이유는 구글링을 하다 보면 클라이언트에서 멱등성 키를 생성하는 방식에 대한 설명이 "클라이언트가 POST 또는 PATCH 요청을 보낼 때마다 UUID를 생성하고 이를 요청에 포함시키는 방식"이라고 되어 있어 마치 매번 요청을 보낼 때마다 새로운 UUID를 보내고, 서버에서 DB 또는 캐시 저장소에 저장하고 비교하여 처리를 해준다고 생각했고, 예시가 UUID밖에 없는 이유는 예측 불가능한 고유성 있는 식별자가 필요한 것이라고 너무 한쪽으로 치우쳐 판단하여 멱등성의 뜻과 다르게 생각하고 있었다.

멱등성의 본래 의미에 맞게, 클라이언트에서 멱등성 키를 UUID로 생성하는 이유는 여러 번의 요청이 동일한 결과를 초래하도록 보장하기 위함이다. 즉, 클라이언트가 POST 또는 PATCH 요청을 보낼 때마다 UUID를 생성하고 이를 요청에 포함시키는 방식은 서버와 클라이언트 간의 일관된 상태 유지를 돕는 중요한 메커니즘이다. 이를 통해 중복 요청이 들어왔을 때 서버가 이를 정확히 인식하고, 동일한 결과를 반환할 수 있게 된다. 또한, 분산 시스템에서 발생할 수 있는 요청의 중복을 효과적으로 처리하여 안정적인 서비스를 제공할 수 있는 장점이 있다.

결국, 클라이언트가 UUID를 생성하여 요청에 포함시키는 것은 단순히 고유한 식별자 사용을 넘어서, 멱등성을 보장하는 핵심적인 방식이며, 시스템의 일관성과 안정성을 높이는 중요한 역할을 한다.
